## 1. מטרה

שירות שרץ על המחשב של נחי (Windows 11, תיקיית עבודה `C:\dev\meeting-agent`) וקולט כל אינטראקציה עסקית משני מקורות:

1. **צ'אט טלגרם עם בוט ייעודי ("הסוכן של נחי")** — הודעות טקסט, הודעות קוליות שנחי מקליט, וקבצי אודיו/מסמכים שנחי משתף אליו (כולל הקלטות שהועברו מווצאפ דרך "שיתוף").
2. **תיקיית קליטה מקומית (`INBOX_PATH`)** — קבצי אודיו של הקלטות שיחה מהסמסונג, שמגיעים אוטומטית דרך שרשרת: טלפון ← Autosync ← Google Drive ← Google Drive for Desktop ← תיקייה במחשב. ההקמה מתוארת ב-`phone-sync-setup.md`.

לכל קלט: תמלול ← חילוץ (מה נקבע + משימות) ← כתיבה למאנדיי ← הודעת אישור חזרה בטלגרם.

**המטרה העסקית:** נחי מנהל עשרות אינטראקציות ביום; בלי מערכת — סיכומים, החלטות והתחייבויות נשכחים. **היעד הסופי:** כל סיכום שיחה נוחת כפריט במחלקת "חדר מיון" במאנדיי (עם "מה נקבע" והסיכום המלא בעדכון הפריט), המשימות שנגזרו נפתחות בלוח המשימות, ונחי מקבל אישור מיידי בטלגרם.

## 2. עקרונות-על (מחייבים, לא המלצות)

1. **אפס נפילות.** כל קלט שנכנס — או שמעובד עד הסוף, או שנשמר ב-`failed/` עם הודעת שגיאה בטלגרם. שום דבר לא נעלם בשקט.
2. **לפני כל כתיבה למאנדיי — קריאת מבנה חי.** תמיד `get_board_info` לפני `create_item` או `change_item_column_values`. לא מנחשים מזהי עמודות או תוויות סטטוס מהזיכרון.
3. **משימות אטומיות.** כל משימה מתחילה בפועל ומכילה פעולה אחת בלבד.
4. **תאריכים מוחלטים בלבד.** "מחר" ← תאריך מפורש, מחושב לפי מועד ההקלטה/ההודעה.
5. **מפתחות ב-`.env` בלבד.** `.gitignore` כולל לפחות: `.env`, `inbox/`, `processed/`, `failed/`, `logs/`, `state.json`. שום מפתח לא בקוד ולא עולה לגיטהאב.
6. **עברית** בכל טקסט שמופנה לנחי (אישורים, שגיאות, שמות פריטים במאנדיי).

## 3. ארכיטקטורה

שירות Python יחיד (`main.py`) עם שני מאזינים במקביל:

**מאזין א' — טלגרם (Bot API רשמי):**
- Long polling: `getUpdates` עם `offset` ו-`timeout` (למשל 50 שניות). אין Webhook, אין URL ציבורי, אין שרת.
- **אבטחה — קריטי:** מעבדים אך ורק הודעות שבהן `from.id == NAHI_TELEGRAM_ID` (מתוך `.env`). כל הודעה אחרת — התעלמות + שורת לוג. שם המשתמש של בוט ניתן לגילוי, ולכן הסינון הזה הוא חובה.
- `CHAT_ID` של הצ'אט נלמד אוטומטית מההודעה הראשונה של נחי (`/start` או כל הודעה), נשמר ב-`state.json` ומאושר מולו פעם אחת.
- סוגי הודעות לטיפול: `text`; `voice` (הקלטה שנעשתה בתוך טלגרם); `audio` ו-`document` עם MIME של אודיו (קבצים ששותפו, כולל הקלטות מווצאפ). `caption` של קובץ נלקח כהקשר לזיהוי הלקוח.
- הורדת מדיה: `getFile` ← הורדה מ-`api.telegram.org/file/bot<TOKEN>/<file_path>`. מגבלת Bot API להורדת קובץ: ‏~20MB. קובץ גדול מזה ← הודעה לנחי: "הקובץ גדול מדי לערוץ הזה — שים אותו בתיקיית הסנכרון" + רישום ב-failed. (הקלטות ווצאפ כמעט תמיד קטנות בהרבה.)
- ניהול `offset` ב-`state.json`. **מגבלה ידועה:** טלגרם שומר עדכונים שלא נמשכו עד ~24 שעות. לכן, בעליית השירות: אם עברו יותר מ-20 שעות מהריצה האחרונה — שלח לנחי: "הייתי כבוי {X} שעות. אם שלחת משהו לפני יותר מיממה — שלח אותו שוב."
- **אמת את פרטי ה-API מול core.telegram.org/bots/api בזמן הבנייה** — אל תסתמך על המפרט הזה לפורמטים המדויקים.

**מאזין ב' — תיקיית קליטה מקומית:**
- הנתיב במשתנה `INBOX_PATH` ב-`.env`. אצל נחי הוא יצביע על התיקייה המקומית של Google Drive for Desktop שמסונכרנת מהטלפון (למשל `...\My Drive\CallRecordings`). ברירת מחדל לפיתוח: `inbox/`.
- `watchdog` על התיקייה: כל קובץ אודיו חדש (`m4a`, `mp3`, `opus`, `wav`, `amr`) נכנס לצינור העיבוד.
- לחכות שהקובץ סיים להיכתב/להסתנכרן (גודל יציב בשתי בדיקות עוקבות) לפני עיבוד.
- **קריאה בלבד — קריטי:** התיקייה ממוראת מול הטלפון והענן. אסור להזיז, לשנות או למחוק בה קבצים. במקום זה: העתקה ל-`processed/YYYY-MM/` ורישום ב-`state.json` (שם קובץ + גודל + חותמת) למניעת עיבוד כפול.
- שמות קבצים בעברית (שם איש הקשר) הם הנורמה — טיפול תקין ב-UTF-8 בנתיבים.

**צינור עיבוד משותף:**
1. אודיו ← תמלול: Groq API, מודל `whisper-large-v3`, פרמטר `language=he`. ‏ffmpeg מותקן וזמין להמרות פורמט. (טקסט מדלג על השלב.)
2. תמליל ← חילוץ מובנה: קריאה למנוע ה-AI עם הפרומפט מסעיף 5 ← JSON.
3. **שער אישור:** ערוץ "שיחה" (הקלטת שיחה) — לא כותבים למאנדיי מיד: שולחים לנחי תצוגה מקדימה עם כפתורי אישור וממתינים להכרעתו (סעיף 7א). שאר הערוצים (תזכורת/שיתוף/הודעה) — ממשיכים ישירות.
4. כתיבה למאנדיי לפי סעיף 6 (בערוץ "שיחה" — רק לאחר אישור).
5. הודעת אישור סופית בטלגרם (`sendMessage` / עריכת הודעת ההמתנה) לפי סעיף 7.
6. תיעוד סיום: קובץ מטלגרם מועבר ל-`processed/YYYY-MM/`; קובץ מתיקיית הסנכרון **מועתק** לשם בלבד (המקור נשאר) ונרשם ב-`state.json`. בשני המקרים — שורת לוג ב-`logs/YYYY-MM-DD.log`.

**זיהוי ערוץ (לשדה "ערוץ"):** קובץ מ-`INBOX_PATH` ← "שיחה"; ‏`voice` בטלגרם ← "תזכורת"; ‏`audio`/`document` בטלגרם ← "שיתוף"; ‏`text` ← "הודעה".

### מנוע ה-AI — סדר עדיפויות
- **עדיפות 1:** `claude -p` (מצב headless על מנוי ה-Max של נחי, **בלי** `--bare` כדי שהאימות הקיים ו-MCP של מאנדיי ייטענו). להריץ מתיקיית הפרויקט, עם `--output-format json` ו-`--allowedTools` שמאשר מראש את כלי המאנדיי. במסלול זה קלוד מבצע גם חילוץ וגם כתיבה למאנדיי ומחזיר JSON של תוצאות.
- **עדיפות 2 (fallback):** `claude -p` רק לחילוץ ה-JSON, וכתיבה למאנדיי דרך ה-REST API ‏(`MONDAY_API_TOKEN` מ-`.env`).
- בדוק את מסלול 1 בפועל בזמן הבנייה; לא יציב — עבור ל-2 בלי להתלבט. יציבות מעל אלגנטיות.

## 5. פרומפט החילוץ (לשמור כ-`prompt.md` ולטעון ממנו)

- הקלט: תמליל/טקסט בעברית + תאריך ושעה, שם לקוח אם זוהה, ערוץ המקור.
- הפלט: **JSON בלבד**, בסכמה:

```json
{
  "client": "שם הלקוח או 'לא משויך'",
  "channel": "שיחה | תזכורת | שיתוף | הודעה",
  "summary_short": "עד 3 שורות בעברית",
  "summary_full": "סיכום מלא ומסודר בעברית",
  "decisions": ["מה נקבע/סוכם/הוחלט — כל הסכמה, מחיר, מועד או תנאי שסוכם מול הלקוח"],
  "tasks": [
    {
      "title": "מתחיל בפועל, פעולה אחת",
      "due_date": "YYYY-MM-DD או null",
      "priority": "דחוף | גבוה | רגיל",
      "is_commitment": true,
      "context": "משפט הקשר קצר"
    }
  ],
  "followups": ["נושאים פתוחים שדורשים מעקב"],
  "open_questions": ["שאלות שנשארו ללא מענה"]
}
```

- כללים בתוך הפרומפט:
  - **`decisions` הוא הלב של הסיכום:** כל דבר שנקבע (מחיר, מועד פגישה, תנאי עסקה, הסכמה עקרונית) נכנס לשם — גם אם אינו מוליד משימה. נקבע גם מועד/פעולה בעקבותיו ← מופיע גם ב-`decisions` וגם כמשימה.
  - התחייבות של נחי ללקוח ("אני אשלח", "אני אבדוק", "אחזור אליך") ⇐ תמיד משימה עם `is_commitment=true`.
  - אין משימות ⇐ מערך ריק, לא להמציא.
  - תאריכים יחסיים מומרים למוחלטים לפי מועד ההקלטה.
  - כותרת משימה כוללת את שם הלקוח כשידוע.

## 6. מאנדיי — מבנה יעד

**לפני הכל:** שליפת רשימת לוחות חיה (`get_user_context` / `search`) ואישור חד-פעמי מול נחי על שני הלוחות. לא להניח מזהים.

### א) הסיכומים — לוח "חדר מיון" (הלוח הקיים של נחי)

- אתר בשליפה החיה את הלוח ששמו מכיל "חדר מיון" ואשר את זהותו מול נחי פעם אחת. **לא נמצא — עצור ושאל.** אין ליצור לוח חדש על דעת עצמך.
- בתוך הלוח: ודא קיום קבוצה ייעודית **"🎙️ סיכומי שיחות"** — אין? צור, באישור נחי.
- כל אינטראקציה = `create_item` בקבוצה זו. מפה את הנתונים לעמודות **הקיימות** בלוח (לפי `get_board_info`): תאריך ושעה, לקוח, ערוץ (שיחה/תזכורת/שיתוף/הודעה), סיכום קצר, מקור. עמודה חיונית חסרה? הוסף רק באישור מפורש של נחי — זה לוח חי ופעיל שלו.
- כ-**Update** על הפריט (`create_update`; ‏HTML בסיסי נתמך — `<b>`, `<ul>`, `<li>`) נכתבים בסדר הזה: **מה נקבע** (`decisions`), הסיכום המלא, ורשימת המשימות שנוצרו.

### ב) משימות — בלוח המשימות הקיים של נחי

- אתר את לוח ניהול המשימות ואשר מול נחי בבנייה הראשונה בלבד.
- ודא קיום קבוצה **"📥 נכנס מהסוכן"** — אין? צור.
- כל משימה = `create_item` בקבוצה, עם מיפוי לעמודות הקיימות (לפי `get_board_info`): תאריך יעד, עדיפות, סטטוס התחלתי.
- משימת התחייבות ← 🤝 בתחילת שם הפריט.
- בעדכון של המשימה: הלקוח, ההקשר, וקישור לפריט היומן.

### כללי פורמט לערכי עמודות (מקור התקלות מספר 1)
- `columnValues` הוא **מחרוזת JSON**, לא אובייקט.
- סטטוס: `{"label": "..."}` עם התווית המדויקת מ-`get_board_info` (תלוית רישיות).
- תאריך: `{"date": "YYYY-MM-DD"}`. טקסט: מחרוזת. מספר: מחרוזת של המספר.
- בספק — `get_column_type_info`.

## 7. הודעת האישור בטלגרם (תבנית)

```
✅ נקלט: {ערוץ} — {לקוח} ({משך אם ידוע})

📋 {סיכום קצר}

📝 מה נקבע:
• {החלטה/הסכמה}
• ...

📌 נוצרו {N} משימות:
1. {כותרת} — יעד: {תאריך}
2. ...

🤝 מסומנות כהתחייבות: {אם יש}
⚠️ {אזהרות: לקוח לא זוהה / תאריך לא נאמר}
```

- אין החלטות ⇐ להשמיט את בלוק "📝 מה נקבע".
- אין משימות ⇐ "📌 לא זוהו משימות לביצוע."
- שגיאה ⇐ "❌ תקלה בעיבוד {שם}. הקובץ נשמר בצד ולא ילך לאיבוד. סיבה: {משפט אחד}."

ההודעה הזו היא הזיכרון החיצוני של נחי — נשלחת תמיד, גם בכישלון.

## 7א. שער האישור להקלטות שיחה (ערוץ "שיחה" בלבד)

- לאחר החילוץ, במקום כתיבה למאנדיי: שלח לנחי הודעת **תצוגה מקדימה** — אותה תבנית מסעיף 7, עם כותרת "🕓 ממתין לאישורך" ובלי שורת "נוצרו X משימות" (במקומה: "📌 משימות מוצעות") — ומתחתיה מקלדת inline עם שני כפתורים: **"✅ אשר ושלח למאנדיי"** ו-**"❌ דחה"**.
- **ניהול ממתינים:** כל פריט ממתין נשמר ב-`state.json` (מזהה, ה-JSON המלא, הפניה לקובץ, `message_id`) — כך שהוא שורד גם הפעלה מחדש של השירות.
- **אבטחת callback — חובה:** מעבדים `callback_query` רק אם `from.id == NAHI_TELEGRAM_ID`; לכל callback עונים ב-`answerCallbackQuery`.
- **אישור** ← כתיבה מלאה למאנדיי (פריט בחדר מיון + המשימות) ← עריכת ההודעה: "✅ אושר ונכתב למאנדיי" + פירוט מה נוצר.
- **דחייה** ← לא כותבים כלום למאנדיי; עריכת ההודעה: "❌ נדחה — לא נכתב למאנדיי. הקובץ שמור."; שורת לוג. הקובץ נשאר ב-`processed/`.
- **תיקון:** אם נחי משיב (Reply) להודעת ההמתנה בטקסט חופשי — הרץ את החילוץ מחדש עם התיקון שלו כהנחיה נוספת, שלח תצוגה מקדימה מעודכנת, וסמן את הקודמת כבוטלה.
- **אנטי-שכחה:** פריט ממתין לעולם אינו פג. עברו 4 שעות בלי הכרעה — תזכורת אחת. בכל עליית שירות, אם יש ממתינים — הודעת ריכוז: "⏳ ממתינים לאישורך: {N} סיכומים".
- ערוצים אחרים אינם עוברים דרך השער. הרחבה עתידית: דגל `APPROVAL_CHANNELS` ב-`.env`.

## 9. סדר בנייה (בנה ובדוק שלב-שלב, מול נחי)

1. שלד פרויקט + `.env` + `.gitignore` + חיבור לבוט: אחרי שנחי שולח `/start`, למד את `CHAT_ID` ושלח "הסוכן חי ✅".
2. קליטת הודעת **טקסט** ← אישור חוזר (בלי מאנדיי עדיין).
3. הודעה **קולית** ← תמלול ← החזרת התמליל לצ'אט.
4. חילוץ JSON מהתמליל.
5. כתיבה למאנדיי (חדר מיון + משימות) — כולל איתור הלוח ויצירת הקבוצה אם חסרה.
6. שער האישור (סעיף 7א): תצוגה מקדימה עם כפתורים ← אישור ← כתיבה; דחייה; תיקון ב-Reply; תזכורת.
7. מאזין `watchdog` על `INBOX_PATH` — בדיקה עם קובץ הקלטת שיחה אמיתי, כולל מעבר מלא בשער האישור.
8. בדיקת קצה-לקצה מלאה + `start-agent.bat`.

בכל שלב: בדיקה אמיתית לפני מעבר הלאה. שלב נכשל — לא ממשיכים.

## 10. מה לשאול את נחי בזמן הבנייה (ורק את זה)

1. ארבעת הערכים: `BOT_TOKEN`, ‏`NAHI_TELEGRAM_ID`, מפתח Groq, טוקן מאנדיי.
2. אישור זהות לוח "חדר מיון" והקבוצה בתוכו, ואישור לוח המשימות.
3. אישור `CHAT_ID` שנלמד מההודעה הראשונה.
